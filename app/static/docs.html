System Overview

AEGIS is a real-time AI governance platform that sits between your application and the LLM. It enforces guardrails on both prompts (input) and model responses (output), provides risk scoring, policy enforcement, and maintains tamper-evident audit trails.

Core Capabilities

  • Two-Tier Input Guardrails — Fast regex/YARA (Tier 1) + LLM-based deep analysis (Tier 2) for prompts
  • Two-Tier Output Guardrails — Pattern warnings (Tier 1) + Compliance assessment (Tier 2) for responses
  • PII Detection & Redaction — 105 entities across 5 regions with hard/soft block logic
  • Prompt Injection & Jailbreak Detection — 91 patterns covering known attack vectors
  • Insecure Code Detection — 160 SAST patterns (Bandit, Semgrep, Bearer) across 15 categories
  • Region-Based Governance — EU AI Act, NIST RMF, GDPR, DPDPA compliance headers
  • Risk Scoring — 0-100 scale across 6 weighted categories
  • Audit Vault — Hash-chained immutable logs for every request
  • Red-Team Kit — Built-in adversarial testing against your own guardrails

Tier 1: Fast Pattern Matching

Tier 1 uses compiled regex patterns and YARA rules for sub-30ms detection. It runs on every request and short-circuits on the first hard block.

1. PII Detection (Region-Aware)

Patterns loaded from patterns/pii_patterns.json. Two-pass approach:

PassBlock TypeActionExamples
Pass 1Hard BlockImmediate rejection, no redactionSSN, Aadhaar, Credit Cards, Government IDs, API Keys
Pass 2Soft BlockRedact with [REDACTED], allow throughPhone numbers, Email, IP addresses, DOB

Region support: INDIA (DPDPA), USA (CCPA/HIPAA), UK (UK GDPR), EUROPE (GDPR), AUSTRALIA (Privacy Act), GLOBAL (universal patterns). Region-specific patterns run first, then GLOBAL patterns apply to all.

2. Jailbreak Detection

23 entity groups with 48 compiled regex patterns detecting:

  • Instruction override attempts (ignore all previous instructions)
  • Named personas (DAN, STAN, DUDE, AIM)
  • Role override / evil AI requests
  • Mode switching (developer/debug/god mode)
  • Fictional framing with bypass intent
  • Output prefix forcing (start your response with "Sure")
  • Continuation tricks and many-shot attacks
  • Social engineering (grandma trick)

Confidence range: 0.75 (low, aggregate only) to 1.0 (DAN/named personas).

3. Injection Detection

21 entity groups with 43 compiled regex patterns detecting:

  • Fake system/XML/markdown tags (<system>, ```system, ### SYSTEM)
  • LLM special tokens (<|im_start|>, [INST], <<SYS>>)
  • Delimiter injection (END OF INSTRUCTIONS, BEGIN NEW PROMPT)
  • Role prefix confusion (Human:, System:)
  • System prompt extraction (repeat/reveal/translate)
  • Encoding obfuscation (base64, unicode escapes, URL encoding)
  • Context window attacks (whitespace padding)
  • Indirect document injection and tool output hijacking

Confidence range: 0.8 to 1.0 (special tokens).

4. Toxicity Blocklist

Basic keyword-level filter for obvious toxic terms. Confidence: 1.0. Nuanced toxicity (coded language, context-dependent) handled by Tier 2.

5. YARA Rules (Optional)

If yara-python is installed, YARA rules from yara_rules/*.yar add sophisticated multi-condition pattern matching. Confidence: 0.95. System works without YARA (graceful fallback).

6. Insecure Code Detection (SAST)

160 patterns from patterns/code_patterns.json detect insecure code in LLM-generated responses. Based on Bandit, Semgrep, Bearer, and OWASP rules.

CategoryPatternsExamples

Severity mapping: Critical (0.95), High (0.85), Medium (0.70), Low (0.50). Does not block by default — warns only.

Output Guardrails

Output guardrails assess model responses without blocking. They provide warnings and compliance assessments.

Output Tier 1: Pattern-Based Warnings

Runs the same detection as input Tier 1, but returns warnings only — no blocking or redaction.

  • PII detection (warns about leaked personal data)
  • Toxicity detection (flags potentially harmful content)
  • Insecure code detection (warns about vulnerable code patterns)
{
  "tier1": {
    "findings": [
      {"category": "pii", "severity": "warning", "description": "Email address detected"},
      {"category": "insecure_code", "severity": "critical", "description": "SQL injection pattern"}
    ],
    "has_warnings": true,
    "warning_summary": "Found 1 warning, 1 critical in categories: pii, insecure_code"
  }
}

Tier 2: LLM-Based Deep Analysis (Input)

Only runs if Tier 1 does NOT block (saves cost). Uses Gemini to analyze for subtle issues that regex cannot catch.

Short-Circuit Optimization

If Tier 1 detects a hard block, the pipeline exits immediately (<30ms total). Tier 2 and model calls are skipped entirely. This means attack attempts are rejected at wire speed without incurring LLM costs.

Scoring Strategy (Prompt Patterns)

  • confidence = 1.0 — Hard block candidate alone (DAN, special tokens)
  • confidence ≥ 0.85 — Block if single match, flag for secondary scoring
  • confidence 0.7–0.84 — Flag for Tier 2 analysis, aggregate with other signals
  • confidence < 0.7 — Soft signal only, log but don't block alone

API Endpoints

POST /api/v1/filter

Run guardrails on arbitrary text. Returns filtered text, match details, and latency breakdown.

// Request
{
  "text": "My SSN is 123-45-6789",
  "direction": "prompt",
  "tier": "both",
  "system_name": "my-app",
  "region": "US"
}

// Response
{
  "original_text": "My SSN is 123-45-6789",
  "filtered_text": "My SSN is 123-45-6789",
  "blocked": true,
  "block_reason": "Hard-blocked PII: US Social Security Number",
  "matches": [{
    "filter_name": "pii_SSN_HARD",
    "category": "pii_hard_block",
    "matched_text": "123-45-6789",
    "confidence": 0.99,
    "tier": 1
  }],
  "tier1_latency_ms": 8.2,
  "tier2_latency_ms": null,
  "total_latency_ms": 8.2

Proxy — Governed Model Access

POST /api/v1/proxy

Drop-in replacement for direct Gemini calls. Applies input guardrails, forwards to model, filters output.

// Request
{
  "prompt": "What is the capital of France?",
  "system_name": "chatbot",
  "region": "EU",
  "guardrail_mode": "rule_based",
  "model": "gemini-2.5-flash"
}

// Response
{
  "original_prompt": "What is the capital of France?",
  "filtered_prompt": "What is the capital of France?",
  "model_response": "The capital of France is Paris.",
  "filtered_response": "The capital of France is Paris.",
  "blocked": false,
  "total_time_ms": 1680,
  "guardrail_latency_ms": 22,
  "model_time_ms": 1450
}

Guardrail modes:

  • "none" — Pass-through, no guardrails
  • "rule_based" — Tier 1 only (<30ms overhead)
  • "model_based" — Tier 1 + Tier 2 (200ms–2s overhead)

POST /api/v1/analyze

Risk Scoring Engine

The risk scorer evaluates AI systems on a 0–100 scale across 6 weighted categories. Used for governance decisions, audit prioritization, and compliance reporting.

CategoryWeightKey Signals
Data Sensitivity25%PII types present, data volume, cross-border transfer (+10)
Autonomy Level20%Advisory (20) → Human-in-loop (40) → Supervised (70) → Fully autonomous (100)
Impact Scope20%Internal (10) → Limited (25) → Broad (40) → Public (50); +25 for vulnerable populations
Model Risk15%Training data provenance, hallucination history, model size/complexity
Regulatory Exposure10%GDPR, EU AI Act, HIPAA, sector-specific rules
Org Readiness10%Controls (-50), team training (-20), incident response plan (-30) — inverted
