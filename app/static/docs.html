End-to-end data flow

Every governed request follows the same logical path. Your application (or the Chat console) sends a prompt; AEGIS evaluates it, optionally calls a model, evaluates the response, and returns a structured allow/deny result with an audit record.

① Client (app, Chat UI, or API integration) │ ▼ ② Ingress Authenticate session · attach actor · select region & mode │ ▼ ③ Fast scan Sensitive data · abuse · policy violations (<30ms typical) │ ├── DENY → entity labels only · no raw secrets returned · audit entry │ ▼ ④ Deep review (advanced mode only) Regional policy · nuanced risk │ ├── DENY → reason + audit · model not called │ ▼ ⑤ Governed inference Sanitized prompt → configured model provider │ ▼ ⑥ Output review Safety & compliance assessment on model text │ ▼ ⑦ Response allow/deny · model text or block reason · audit sealed

Stages at a glance

Ingress

Request accepted

Region, guardrail mode, and provider are resolved. The request is attributed to a user and system for auditing.

Screening

Fast decision

High-risk content is stopped immediately. Clear violations never reach a model — saving cost and reducing exposure.

Inference

Model call

Only approved prompts are forwarded. Compliance context for the selected region is applied before the provider runs.

What you receive on block

When a request is denied, responses include identified entity types (for example, government ID or financial identifier categories) — not the underlying sensitive text. This protects users while giving operators enough context to investigate.

Latency & performance

AEGIS is designed so the common path stays fast and expensive work runs only when needed.

Fail-fast behavior

If screening rejects a prompt, the pipeline stops immediately. Deep review and model inference are skipped, which keeps median latency low and avoids unnecessary provider charges.

Reliability & auditability

Governance only matters if you can trust the record of what happened.

  • Every decision is logged — allow, deny, and filter events write to the audit vault with actor, timestamp, and outcome.
  • Tamper-evident chain — entries are linked so historical records can be verified for integrity.
  • Consistent deny semantics — blocked requests follow the same response shape whether stopped at screening or deep review.
  • Admin telemetry — operators with admin role can view traffic trends, block rates, and PII category summaries over time.
  • Session-based access — API and UI use secure cookies; unauthenticated clients cannot reach governed endpoints.

Regional governance

Select a region when submitting a prompt. AEGIS applies the appropriate compliance framing for India, Europe, USA, China, or Australia before inference and during output assessment.

API overview

Integrate via REST. Full request and response schemas are in Swagger and ReDoc.

POST /api/v1/analyze

Full governed flow: screen prompt, optional deep review, call model, assess output. Returns allow, response, pipeline timings, and tier summaries. 

{
  "prompt": "Your user message",
  "region": "india",
  "guardrail_mode": "basic",
  "inference_provider": "gemini",
  "model": "gemini-2.0-flash"
}
POST /api/v1/filter

Screen text only — no model call. Use when you need governance checks inside your own inference stack.

POST /api/v1/auth/login GET /api/v1/auth/me

Authenticate for session cookies used by the Chat UI and API.
